# InvoiceHub 고도화 로드맵 (v3)

> MVP(Phase 0~4, T001~T405)와 v2 고도화(Phase 5~8, T501~T803) 완료 이후의 **v3 고도화 단계** 로드맵입니다.
> 이전 개발 히스토리는 아카이브를 참조하세요.
> - MVP: `docs/roadmaps/ROADMAP_v1.md` (F001~F006)
> - v2 고도화: `docs/roadmaps/ROADMAP_v2.md` (F007~F009)

## 개요

MVP는 견적서 웹 조회(F001)·PDF 다운로드(F002)·Notion 연동(F003)·어드민 목록(F004)·공유 URL(F005)·어드민 인증(F006)을 갖춘 상태로 배포되었고, v2에서 어드민 전용 레이아웃+세션 지속(F007)·공유 링크 클립보드 복사(F008)·다크모드 커버리지(F009)까지 완성되었습니다.

이번 v3는 **견적서 목록의 관리 효율**과 **링크 공유 경험**을 강화합니다. 견적서 건수가 늘어나면서 드러난 목록 탐색의 한계와, 클립보드 복사에 머물러 있던 공유 수단을 확장하는 것이 목표입니다. 사용자가 명시적으로 요청한 3가지가 이번 스코프입니다.

- **견적서 목록 고급 조회 (요구사항 1 / F010)**: 검색·상태 필터·날짜 범위 필터·정렬·페이지네이션·검색어 하이라이팅
- **고유 링크 생성 및 표시 시스템 (요구사항 2 / F011) — 최우선**: 목록에 링크 컬럼 추가, (선택) 짧은 URL, (선택) QR 코드 생성
- **링크 공유 통합 (요구사항 3 / F012)**: 이메일(mailto)·메신저 공유 버튼, 링크 미리보기 메타데이터(OG 태그), (결정 필요) 링크 통계 추적

### 코드베이스 현황 진단 (이번 스코프의 근거)

로드맵을 실제 갭에 맞추기 위해 착수 전 코드 상태를 조사한 결과입니다. **이미 되어 있는 것을 다시 만들지 않도록** 태스크를 설계했습니다.

| 영역 | 현재 상태 | 갭 / 조치 방향 |
|------|-----------|----------------|
| 목록 검색/필터/정렬 | `InvoiceTable.tsx`는 `getAllInvoices()` 결과(발행일 내림차순)를 **전량 한 번에 렌더**. 검색·필터·정렬·페이지네이션 UI 전무 | 클라이언트 측 검색/필터/정렬/페이지네이션 도입 (서버 데이터는 그대로, 뷰 레이어에서 가공) |
| 목록 데이터 조회 | `getAllInvoices()`는 발행일 `descending` 정렬만 수행, 검색/필터 파라미터 없음 (`src/lib/notion.ts`) | 목록 규모가 작으므로 클라이언트 가공을 우선. 대량화 시 서버 필터로 확장(백로그) |
| 링크 표시 | 목록 마지막 열은 "미리보기"(Eye)·"복사"(Copy) 버튼뿐. **URL을 텍스트로 보여주는 링크 컬럼 없음** (`InvoiceTable.tsx`) | 링크 컬럼 신설 — URL 표시 + 기존 복사/미리보기 버튼 통합 배치 |
| 공유 URL 생성 | `buildInvoiceShareUrl(token)` 존재 (`src/lib/invoice-url.ts`). 클립보드 복사 동작(F008) 완비 | 재사용. 짧은 URL/QR은 이 유틸 위에 확장 |
| 짧은 URL | 미지원 (풀 UUID 토큰 URL만) | **선택 확장** — 자체 단축은 매핑 저장소 필요 → 결정 이슈와 연동 |
| QR 코드 | 미지원 | **선택 확장** — 클라이언트 측 생성 라이브러리(신규 의존성) 필요 |
| 링크 미리보기(OG) | 루트 `layout.tsx`에 **정적** OG 태그(사이트 공통)만 존재. `invoices/[token]/page.tsx`에 `generateMetadata` 없음 → 견적서별 동적 미리보기 부재 | 공개 조회 라우트에 `generateMetadata` 추가 (견적서 번호/클라이언트 반영) |
| 이메일/메신저 공유 | 없음 | mailto 링크 + 카카오톡/텔레그램 등 공유 URL 스킴 버튼 신설 |
| 링크 통계(조회수) | Notion 스키마에 **카운터 필드 없음**. 저장 위치 미결정 | **착수 전 결정 필요** (아래 "결정 필요 항목" 참조) — 태스크로 확정하지 않음 |

F012의 통계 추적은 아키텍처 결정이 선행되어야 하므로 가장 뒤에 둡니다.

## 개발 워크플로우

1. **작업 계획** — 기존 코드베이스 상태를 파악하고 `ROADMAP.md`를 갱신
2. **작업 생성** — 태스크 단위로 분해하고 관련 기능 ID(F001~F012)와 매핑
3. **작업 구현** — 명세를 따라 구현하고, ⚠️ **구현 완료 후 반드시 Playwright MCP로 테스트 수행 (테스트 없이 완료 처리 금지)**
   - Playwright MCP 절차: `browser_navigate` → `browser_snapshot` → `browser_fill_form`/`browser_click` → `browser_network_requests` → `browser_snapshot` → 실패 시 `browser_console_messages` 확인 후 수정 → 재테스트
4. **로드맵 업데이트** — 완료된 태스크를 `[x]`로 표시

### Phase 10: 고유 링크 생성 및 표시 시스템 — 최우선 (F011)

> Phase 9에서 정비한 목록 테이블에 **링크 컬럼**을 추가하고, 클라이언트 전달용 공개 URL을 목록에서 바로 확인할 수 있게 합니다. 짧은 URL과 QR 코드는 **선택 확장**으로, 필수 컬럼 완성 이후 별도 판단합니다.

**필수 태스크**

| 태스크 ID | 설명 | 관련 기능 | 상태 |
|-----------|------|-----------|------|
| T1001 | 링크 컬럼 신설 — 각 행에 공개 URL을 텍스트로 표시(말줄임+`title`), 기존 복사/미리보기 버튼을 이 컬럼으로 통합. `buildInvoiceShareUrl` 재사용 | F011, F005 | `[x]` |
| T1002 | 링크 표시 활성 조건 — `승인` + 공유토큰 보유 행만 실제 링크 노출, 그 외에는 "발행 전" 등 비활성 안내(기존 `canCopy` 로직 재사용) | F011 | `[x]` |
| T1003 | 링크 컬럼 반응형 — 좁은 화면에서 URL 말줄임/가로 스크롤 처리, 복사·미리보기 버튼과 정렬 유지(`overflow-x-auto` 테이블과 정합) | F011, F009 | `[x]` |

**선택 확장 (우선순위 낮음 — 필수 컬럼 완성 후 개별 결정)**

|-----------|------|-----------|------|
| T1051 | (선택) QR 코드 생성 — 링크의 QR을 클라이언트 측에서 렌더링(모달/팝오버), 다운로드 버튼 포함. **신규 의존성(`qrcode` 등) 추가 결정 필요** | F011 | `[ ]` |

> 선택 확장은 신규 의존성/저장소 도입을 수반하므로 로드맵에서 확정하지 않습니다. 필수 링크 컬럼(T1001~T1003)이 v3의 최우선 완성 대상이며, 선택 항목은 완료 후 별도 승인 시 착수합니다.

**Phase 10 테스트 체크리스트 (⚠️ 구현 후 필수 수행 — Playwright MCP)**
- `[x]` 승인 상태 행의 링크 컬럼에 정확한 공개 URL(`{BASE_URL}/invoices/{토큰}`)이 표시되는지 검증 (`browser_snapshot`)
- `[x]` 비승인/토큰 없는 행은 링크가 비활성 안내로 표시되는지 검증 — 실 데이터가 승인 1건뿐이라 코드 리뷰(삼항 분기)로 대체 확인
- `[x]` 링크 컬럼에서 복사/미리보기 버튼이 여전히 정상 동작하는지 검증 (`browser_click`, `browser_evaluate`로 클립보드 내용 확인)
- `[x]` 좁은/넓은 뷰포트에서 링크 컬럼이 레이아웃을 깨지 않고 표시되는지 검증 (`browser_resize`, `browser_take_screenshot`, 라이트/다크 모두 확인)
- `[ ]` (선택 구현 시) QR 코드가 링크와 일치하고 다운로드되는지 검증 — T1051 미착수로 해당 없음

## 결정 필요 항목 (착수 전 확정)

> 아래 항목들은 **새 백엔드 의존성/저장소 도입**이라는 큰 결정을 수반하므로 로드맵에서 태스크로 확정하지 않고 별도로 남깁니다. 결정이 내려지면 해당 Phase에 태스크로 편입합니다.

### D1. 링크 통계 추적 (조회수 등) — F012 관련

Notion 스키마에 조회수/카운터 필드가 없어 저장 위치를 먼저 정해야 합니다. 대안:

| 대안 | 방식 | 장점 | 단점 |
|------|------|------|------|
| (a) Notion 속성 증분 | Invoices DB에 `조회수` Number 속성 신설, 매 공개 조회마다 `pages.update`로 +1 | 신규 인프라 없음, 데이터 일원화 | Notion write API 쿼터 소모, 동시 조회 시 레이스 컨디션, 조회당 쓰기로 느림 |
| (b) 경량 KV 카운터 | Vercel KV / Upstash Redis에 토큰별 카운터 저장 | 원자적 증분(빠름·정확), Notion 부하 없음 | **신규 백엔드 의존성 추가**, 별도 계정/환경변수/과금 |
| (c) 외부 애널리틱스 | Vercel Web Analytics / Plausible 등 페이지뷰 집계 | 구현 최소, 유지보수 적음 | 견적서별 커스텀 집계·표시 제약, 어드민 목록 내 노출 어려움 |

→ **결정 필요**: 통계의 정확도 요구 수준과 인프라 추가 허용 여부에 따라 선택. 새 의존성 추가는 로드맵에서 확정하지 않음.

### D2. 짧은 URL 형식 (선택) — F011 / T1052 관련

- 자체 단축: 단축 코드 ↔ 공유토큰 매핑 저장소 필요 → **D1과 동일한 저장소 결정 이슈**와 연동.
- 외부 단축 서비스: 외부 API 의존·링크 영속성/추적 우려.
- → 저장소 결정(D1) 이후 함께 판단. 결정 전까지 T1052 미착수.

### D3. QR 코드 라이브러리 (선택) — F011 / T1051 관련

- 클라이언트 측 생성 라이브러리(`qrcode`, `qrcode.react` 등) **신규 의존성 추가** 필요. 상대적으로 경량이나 도입 여부는 승인 필요.
- → 필수 링크 컬럼 완성 후 사용자 승인 시 착수.

### D4. 카카오톡 공유 SDK — F012 / T1102 관련

- 카카오 공유는 JavaScript SDK + 앱 키 등록이 필요할 수 있음. 우선 무의존 수단(텔레그램 공유 URL, `navigator.share`)부터 구현하고, 카카오 SDK 도입은 별도 결정.
